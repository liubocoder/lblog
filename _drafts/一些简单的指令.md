#git 常用的指令
本地执行的命令

git clone <url> 获取远端分支
git checkout -b xxx 创建并切换分支   git checkout -b nnn origin/xxx
git branch -a  所有分支  -b 创建分支  -d删除分支
git merge xxx 合并本地分支xxx
git commit -m MM 提交日志
git push <仓库名> <本地分支:远端分支>  将代码push到仓库分支  本地分支不填写则会删除远端分支
git push origin --delete xxx 删除远端分支  xxx为远端分支名 remote/origin/xxx
git fetch <仓库名> <远端分支:本地分支> 获取远端分支代码
git pull <仓库名> <远端分支:本地分支> 获取远端分支并合并
git pull = git fetch + git merge
add xxx  添加修改的文件 下一步一般是执行commit
git reset –hard origin/master 将本地分支回退到远端分支
git diff 查看是否有修改过的文件未提交
git status 查看状态和远端分支对比：是否有未push或pull的提交
git cherry-pick <commit id> 合并某次提交  git cherry-pick是本地特性，本地要有这个commit才可以被git cherry-pick
git push origin 分支名 --force  强行覆盖远端分支提交

关于svn
1.本地修改还原
svn revert [-R] xxx_file_dir
如果需要回滚的是一个目录则加上-R（递归）可选参数

2.回退到已提交的版本
1）执行#svn update命令保证工作区文件是最新的，比如最新版本号是20
2）然后找出要回滚的确切版本号：
执行svn log xxx_file_dir
假设根据svn log日志查出要回滚的版本号是10，如果想要更详细的了解情况，可以使用svn diff -r 20:10 [xxx_file_dir]
3)回滚到版本号10：
执行svn merge -r 20:10 xxx_file_dir
4)提交回滚：
svn commit -m "注释..." 
提交后版本变成了29

java已经封装的aes算法只满足128key,加密后长度是在以16byte[]返回的，因为读取原文的时候是从头开始选取16个字节，然后转换为byte，再根据s盒子开始第一轮加密，所以，密文只会返回32字节，因为会把byte拆分，迭代等。
也就是说，小于16字节的原文会得到32字节长度的密文
大于16字节的，就是16*n长的原文会得到32*n长度的密文
就好比楼主提供的数据，src=123456789123456789123456789长度27，所以得到32*2长度的密文

repo的简单配置（单次使用有效）
1、根目录下创建.bin文件夹
mkdir ~/.bin
2、配置为临时环境变量（当前Shell有效）(也可配置为永久的)
export PATH=~/.bin:$PATH
3、下载repo
git clone https://gerrit-googlesource.lug.ustc.edu.cn/git-repo
4、将git-repo中的repo文件复制到 1 创建的.bin目录中
cd git-repo 
cp repo ~/.bin/
5、修改权限
chmod a+x ~/.bin/repo
6、创建同步源码的工作目录
mkdir android_source
7、在工作目录中创建.repo目录
cd android_source 
mkdir .repo
8、将下载的git-repo拷贝到.repo下,并改名为 repo
9、可以重新初始化了
repo init -u xxxxxxxxx -b xxxx
10、开始同步
repo sync
